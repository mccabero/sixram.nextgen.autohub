## PostgreSQL Schema

### Recommended PostgreSQL Naming Strategy

For the first migration, preserve the source model's PascalCase table and column names through Prisma `@@map` and direct field names. This minimizes API and ETL risk. PostgreSQL will quote those identifiers, which is less idiomatic but safer for an initial parity migration.

After parity, a later cleanup can move the physical database toward lowercase snake_case if desired.

Important naming checkpoint:

- The scaffolded context and active context do not fully agree on physical table names.
- Examples: `Customer`, `Membership`, `Parameter`, `Payment`, `QuickSale`, and other scaffolded tables appear singular in scaffolded mappings, while the active context uses plural DbSet conventions and explicit plural overrides for many tables.
- Before generating the first Prisma migration, validate the live SQL Server table names with `INFORMATION_SCHEMA.TABLES` or SQL Server Management Studio.

- `VehicleModel` has multiple `Parameter` references: body and classification.
- `Deposit`, `Expense`, `PaymentDetail`, and `QuickSale` reference payment-type parameters.

Use relation names such as:

```prisma
AdvisorEstimates   Estimate[] @relation("EstimateAdvisorUser")
EstimatorEstimates Estimate[] @relation("EstimateEstimatorUser")

```

Do not rely on Prisma's automatic relation naming for these tables; it will become noisy and unstable.

## Data Type Mappings

| C# / EF type | SQL Server source | PostgreSQL target | Prisma target | Notes |
| --- | --- | --- | --- | --- |
| `int` | `int` | `integer` | `Int` | Use identity only for `Id`. |
| `int?` | nullable `int` | nullable `integer` | `Int?` | Preserve nullability. |
| `bool` | `bit` | `boolean` | `Boolean` | Convert `0/1` to `false/true`. |
| `decimal` | usually `decimal(18,2)` | `numeric(18,2)` | `Decimal @db.Decimal(18, 2)` | Preserve as Decimal, not JS number. |
| `Membership.PointsEarned` | `decimal(18,0)` | `numeric(18,0)` | `Decimal @db.Decimal(18, 0)` | Special case from scaffolded mapping. |
| `DateTime` | `datetime` | `timestamp without time zone` | `DateTime @db.Timestamp(3)` | Confirm UTC/local semantics before cutover. |
| `DateTime?` | nullable `datetime` | nullable `timestamp without time zone` | `DateTime? @db.Timestamp(3)` | Confirm null behavior in reports. |
| `DateTimeOffset` | `datetimeoffset` | `timestamp with time zone` | `DateTime @db.Timestamptz(3)` | Used by `CameraEvent.EventDateTime`. |
| SQL Server `date` | `date` | `date` | `DateTime @db.Date` | Used by `ProductInventoryCheck.CheckDate`. |
| `string` with max length | `nvarchar(n)` | `varchar(n)` | `String @db.VarChar(n)` | PostgreSQL `varchar` length is character count. |
| `string?` with max length | nullable `nvarchar(n)` | nullable `varchar(n)` | `String? @db.VarChar(n)` | Preserve nullability. |
| unbounded `string` | `nvarchar(max)` or convention text | `text` | `String @db.Text` | Includes raw XML and serialized checklist/details. |
| `ICollection<T>` navigation | EF relationship only | not a column | Prisma relation field | Add in relation pass. |

## MSSQL Compatibility Issues

### Schema Generation Issues

| Issue | Source usage | PostgreSQL/Prisma action |
| --- | --- | --- |
| SQL Server provider | EF uses `Microsoft.EntityFrameworkCore.SqlServer` | Replace with Prisma PostgreSQL provider. |
| PascalCase and singular table names | Mixed scaffolded and active mappings | Validate live names, then use `@@map`. |
| `IDENTITY(1,1)` | SQL Server identity columns | Use Prisma `@default(autoincrement())`. |
| Filtered indexes | `HasFilter("[IsPrimaryCompany] = 1")`, `HasFilter("[IsActive] = 1")` | Use PostgreSQL partial indexes in raw SQL migrations. |
| Runtime DDL | Schema initializers create/alter tables at startup | Replace with versioned Prisma migrations. |
| SQL Server catalog checks | `OBJECT_ID`, `COL_LENGTH`, `sys.*` | Remove from app runtime; migrations own schema state. |
| `MERGE` seed logic | Permission seeding | Use Prisma `upsert` or PostgreSQL `INSERT ... ON CONFLICT`. |
| `SCOPE_IDENTITY()` | SQL Server insert identity retrieval | Use Prisma create return value or PostgreSQL `RETURNING`. |
| `GO` batch separators | SQL sync scripts | Invalid in PostgreSQL. Regenerate migrations. |
| `GETUTCDATE()` defaults | Raw SQL initializers | Use `now()` or application-generated UTC. |
| Include indexes | SQL Server index includes in initializer | PostgreSQL supports `INCLUDE`; raw migration may be required. |

### Data and Behavior Issues

| Issue | Risk | Action |
| --- | --- | --- |
| SQL Server `.bak` file | Cannot restore directly to PostgreSQL | Restore to SQL Server staging, export, transform, load into PostgreSQL. |
| Date/time semantics | `datetime` has no timezone; `datetimeoffset` does | Define UTC policy before ETL. |
| Decimal values | JS `number` can lose precision | Use Prisma `Decimal` and serialize deliberately in API responses. |
| Case sensitivity | PostgreSQL identifiers and string comparisons differ from SQL Server defaults | Normalize comparisons in queries and add lower/functional indexes where needed. |
| Empty string vs null | SQL Server app may treat them loosely | Preserve source values and validate forms/API behavior. |
| Raw SQL reports | SQL Server syntax will fail in PostgreSQL | Rewrite inventory/audit/report SQL using PostgreSQL syntax. |
| `OperationAccessCode.CodeValue` | Sensitive plaintext code may be retained | Review whether to migrate, null out, or remove after cutover. |
| Serialized text fields | `InspectionDetails`, `ChecklistJson`, `RawXml` | Consider `jsonb` for JSON fields only after confirming data is valid JSON. |

## Relationship Plan

After the scalar model is generated, add relation fields for the following FK patterns:

- Customer: deposits, estimates, inspections, invoices, job orders, memberships, payments, quick sales, vehicles.
- Vehicle: inspections, estimates, job orders; references customer, vehicle model, and multiple parameter rows.
- VehicleMake and VehicleModel: make/model hierarchy plus parameter references.
- Catalog: products/services/packages with package product/service bridge tables.
- Inventory: product to inventory transactions and inventory check items.
- Operations: inspection/estimate/job order/invoice aggregates and their line tables.
- Finance: deposits, payments, payment details, quick sales, expenses, petty cash.
- RBAC: users, roles, user roles, permissions, role permissions.

Use explicit relation names wherever an entity has multiple FKs to the same target table.

## Recommended Prisma Migration Order

1. Create `schema.prisma` with generator/datasource and scalar models only.
2. Add `@@map` for every physical table after validating live SQL Server names.
3. Add `@db` native type attributes for string lengths, decimals, dates, and timestamps.
4. Add unique and regular indexes Prisma can express.
5. Add raw SQL migration for PostgreSQL partial indexes.
6. Add relation fields in domain groups.
7. Run `prisma format`.
8. Generate a PostgreSQL dev database from the schema.
9. Perform a SQL Server export to PostgreSQL dry-run.
10. Run row-count, FK, decimal total, and report-total reconciliation.

## Validation Checklist

- Confirm live SQL Server table names for singular/plural differences.
- Confirm all decimal precision and scale from live SQL Server metadata.
- Confirm all `nvarchar(max)` fields that contain JSON are valid JSON before mapping to `jsonb`.
- Confirm timezone policy for all transaction and event timestamps.
- Confirm all active startup schema initializers are represented as migrations.
- Confirm all RBAC seed data is converted to deterministic Prisma seed/upsert logic.
- Confirm `OperationAccessCode` sensitive values should be migrated.
- Confirm all FK constraints and delete behaviors in production before generating relation fields.
- Confirm all existing indexes from production, not only those visible in source.

## Final Recommendation

Generate the Prisma schema from the scalar field inventory first, but do not run production migrations until the live SQL Server schema has been introspected. The source model is mature enough to build a Prisma baseline, but table-name ambiguity and runtime schema initializers make live validation mandatory before cutover.
